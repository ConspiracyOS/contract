## Area 1: `contract` and `spec` CLI commands

### `agent-config contract`

Three subcommands:

**`contract new`** ŌĆö interactive scaffold for a new project contract.
- Prompts: id, description, type (atomic/holistic), trigger (commit/pr/merge), scope (global or glob pattern), first check module (select from list), on_fail.
- Writes `.agent/contracts/<id>.yaml`.
- Validates id is not already in use.

**`contract list`** ŌĆö tabular output of all contracts (builtins + project).
- Columns: id, trigger, scope summary, source (builtin/project).

**`contract check <id>`** ŌĆö run a single contract by id against the current project root.
- Useful during contract authoring to iterate without running the full audit.
- Exits 1 if the contract fails.

### `agent-config spec`

Three subcommands:

**`spec new`** ŌĆö scaffold a new RFC directory.
- Prompts: title (slug auto-derived), brief description.
- Auto-increments RFC number by scanning existing `.agent/specs/RFC-NNN-*/` directories.
- Creates `.agent/specs/RFC-NNN-<slug>/brief.md` and `proposal.md` with structured stubs.
- `brief.md` stub: Problem, Scope, Success criteria, Contract coverage sections.
- `proposal.md` stub: Approach, Acceptance criteria (with concrete examples required), Implementation notes sections.

**`spec list`** ŌĆö lists all RFCs with inferred lifecycle status.
- Status inference: `draft` = brief.md only; `proposed` = proposal.md present; `approved` = approval.md present; `implemented` = review.md present.
- Columns: id, title, status.

**`spec status <id>`** ŌĆö full RFC lifecycle view.
- Shows which files exist and which are missing.
- Prints next required action (e.g. "awaiting approval.md from owner").

## Area 2: Secrets management

### Encryption tool: `age`

[`age`](https://age-encryption.org/) is the chosen encryption tool. Rationale: modern, purpose-built for file encryption, simple passphrase API, well-audited, single binary. Consistent with the project's existing pattern of requiring CLI tools (`gh`, `git`).

### Vault model

```
.agent/vault           # age-encrypted file containing YAML keyŌåÆvalue pairs; committed to git
.vault_password        # plaintext passphrase; gitignored
.vault_password.sh     # optional; stdout used as passphrase (enables `op read` for 1Password)
```

The vault file is decrypted to a YAML document: `{ KEY: value, KEY2: value2 }`.

Passphrase resolution order:
1. If `.vault_password.sh` exists and is executable ŌåÆ run it, use stdout as passphrase
2. Else if `.vault_password` exists ŌåÆ read as passphrase
3. Else error with instructions

### `agent-config vault` subcommand

- `vault get KEY` ŌĆö decrypt vault, print single value to stdout
- `vault set KEY VALUE` ŌĆö decrypt, update/add key, re-encrypt, write back
- `vault list` ŌĆö decrypt, print keys only (no values)
- `vault export` ŌĆö decrypt, print `export KEY=value` lines for shell sourcing

### CI access

Store vault passphrase as a GitHub Actions secret (`VAULT_PASSWORD`). Generated CI workflow includes a setup step:

```yaml
- name: Set up vault
  run: echo "${{ secrets.VAULT_PASSWORD }}" > .vault_password
  if: env.VAULT_PASSWORD != ''
  env:
    VAULT_PASSWORD: ${{ secrets.VAULT_PASSWORD }}
```

### What `init` generates

When the user opts in to secrets management during `agent-config init`:
- Creates `.agent/vault` (age-encrypted empty YAML `{}`)
- Adds `.vault_password`, `.vault_password.sh`, `.agent/vault` to `.gitignore` where missing (vault itself is NOT gitignored ŌĆö it's committed)
- Generates `.env.example` with placeholder key names and comments (no values)
- Prints setup instructions for CI (`VAULT_PASSWORD` secret)

### Built-in contracts

- `C-SEC01`: No `.env` files committed ŌĆö scope `**/.env*`, `path_not_exists` or `no_regex_in_file` on `.gitignore` confirming `.env` is listed.
- `C-SEC02`: No secrets patterns in source ŌĆö `command: run: "git log --all --full-history | head -1"` placeholder; real scan deferred until a lightweight secrets scanner is chosen.
- `C-SEC03`: `.vault_password` and `.vault_password.sh` in `.gitignore` ŌĆö `regex_in_file` on `.gitignore`.

## Area 3: Remaining stack contracts

### Rails (`src/stacks/rails.ts`)

- `C-RB01`: `bundle exec rubocop` passes (command, pr trigger, fail)
- `C-RB02`: `bundle exec rails test` passes (command, pr trigger, fail; skip_if path_not_exists: test) with rspec fallback handled in the run script
- `C-RB03`: No bare `find_by` without nil guard ŌĆö `no_regex_in_file: pattern: "find_by\([^)]+\)\."` scoped to `app/**/*.rb`, `require_exemption`

### Rust (`src/stacks/rust.ts`)

- `C-RS01`: `cargo clippy -- -D warnings` passes (command, commit trigger, fail)
- `C-RS02`: `cargo fmt --check` passes (command, commit trigger, fail)
- `C-RS03`: `proptest::` import present in all `**/*.rs` files excluding `build.rs` and generated files (regex_in_file, pr trigger, require_exemption)

### Mobile (`src/stacks/mobile.ts`)

- `C-MO01`: `npx expo-doctor` passes (command, pr trigger, warn; skip_if path_not_exists: app.json)
- `C-MO02`: No hardcoded `localhost` in non-test source ŌĆö `no_regex_in_file: pattern: "localhost"` scoped to `src/**/*`, exclude `**/*.test.*`, `require_exemption`

### Containers (`src/stacks/containers.ts`)

- `C-CT01`: No host port binding ŌĆö `no_regex_in_file: pattern: '"[0-9]+:[0-9]+"'` scoped to `**/docker-compose*.yml`, `**/compose*.yaml`, fail
- `C-CT03`: No `latest` image tags ŌĆö `no_regex_in_file: pattern: ":latest"` scoped to same compose files, warn
- `C-CT04`: No `localhost` in service env vars ŌĆö `no_regex_in_file: pattern: "localhost"` scoped to compose files, require_exemption

Note: `C-CT02` (all services declare `networks:`) dropped ŌĆö not reliably checkable with regex; deferred to Phase 3 with YAML-aware parsing.

## Area 4: Documentation discipline

### Contracts

**`C-DOC01`**: `docs/ARCHITECTURE.md` must exist at project root.
- `path_exists: path: docs/ARCHITECTURE.md`, commit trigger, fail.
- skip_if: path_not_exists: docs (don't enforce on projects without a docs/ directory).

**`C-DOC02`**: Markdown files in `docs/` root must declare a type header.
- `regex_in_file: pattern: "<!-- type:"` scoped to `docs/*.md`, commit trigger, warn.
- skip_if: path_not_exists: docs.

Both are opt-in at the stack level ŌĆö added as built-in contracts only when the project has a `docs/` directory (detected at init time or enforced via skip_if).

### Staleness detection

Deferred to Phase 3. The spec notes this as a "future investigation." Phase 2 scope is C-DOC01 and C-DOC02 only.

## Area 5: C-PROC04 ŌĆö Require contract coverage

### Approach: warn on uncovered source files

When `require_coverage: true` in `.agent/config.yaml` (the default), the audit engine adds a synthetic post-audit step:

1. Collect all files matching the coverage patterns (default: `src/**/*`, `lib/**/*`, `app/**/*`; configurable via `contracts.coverage_paths` in config).
2. For each file, check whether at least one non-global contract had it in scope during this audit run.
3. Files with zero applicable contracts emit a `warn` result with a synthetic contract id `C-PROC04`.

Behaviour:
- `on_fail: warn` ŌĆö does not exit 1, but appears in audit output.
- Disabled entirely when `require_coverage: false`.
- Only fires for the `commit` trigger (not pr/merge, to avoid noise in CI).

### Config extension

```yaml
contracts:
  audit_on: [commit, pr]
  behavioral_on: [merge]
  require_coverage: true
  coverage_paths:           # optional override
    - src/**/*
    - lib/**/*
```

### Engine changes

- `runAudit()` receives the coverage config.
- After all contracts run, collect the set of files that were evaluated by at least one check.
- Diff against the full set of files matching coverage_paths.
- Append synthetic `CheckResult` entries for uncovered files.
